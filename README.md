# RAG App

## アーキテクチャ選定

RAGアプリのインフラ構成について、運用コスト・性能・将来の拡張性の観点から3案を比較した。

### 要件
- 利用は平日9〜19時に集中
- 少人数の同時利用
- チャット1回の所要時間は60秒超で、LLM生成時間が支配的(SSEストリーミング)
- 瞬間的なスケール性能より、常時コストの最小化を優先

### 比較した構成
- **A案（現行）**：ECS Fargate Spot + RDS MySQL + Chroma on EFS
- **B案（Lambda移植）**：コンテナLambda + Function URL（データ層は現行構成）
- **C案（再設計）**：VPC外Lambda + DynamoDB + S3 Vectors

### 結論
- **B案**：今回採用したChroma(SQLiteバックエンド)構成では同時実行を制限する運用となる事に加え、NATインスタンスなどの固定費も発生するため、今回の要件では採用を見送った。
- **A案とC案**：本試算では約14,400チャット/月を境にコストが逆転する。現在の利用規模（月数百チャット）ではC案の方が低コストだが、データ層移行(MySQL→DynamoDB、Chroma→S3 Vectors)のコストを考慮し、現時点ではA案を採用した。
- **将来計画**：ベクトル層 → 文書取込 → データ層 → コンピュート/ingress の4段階で段階的にC案へ移行できる設計とした。
